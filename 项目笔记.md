# APIWatch 项目笔记

> 面向本地开发态的、多框架 / 未来多语言的 **API 调用观测工具**。接入一行探针，即可在本地看板上看到每个 API 的调用次数、耗时、错误率、最近请求与 trace 信息。

最后更新：2026-07-09

---

## 一、项目相关信息

### 1.1 一句话定位

轻量版、本地开发态的 API Observability 工具——**本地运行、零后端、多框架统一、一行接入**。当前 MVP2 支持 Python ASGI（FastAPI / Litestar）、Flask、Django，架构按「契约先行」设计，未来可平行扩展到更多语言。

### 1.2 核心特性

- **一行接入**：`app.add_middleware(ApiWatchASGIMiddleware)`，无需改动业务代码。
- **本地运行、零后端**：一个本地 collector（`127.0.0.1:8765`）+ SQLite，不依赖任何云服务。
- **不影响业务**：事件 fire-and-forget 异步上报，collector 不可用时静默丢弃，绝不阻塞或拖慢请求。
- **请求级观测**：method / path / route / 状态码 / 耗时 / 错误 / trace_id，聚合出调用次数、平均耗时、P95、错误率、慢接口与错误接口排行。
- **W3C Trace Context**：trace 标识采用标准 `traceparent`，可与 OpenTelemetry 生态衔接，为未来分布式串联铺路。
- **语言中立契约**：探针与 collector 之间只传标准化 JSON，未来加 Go / Node 探针时 collector 与看板一行不改。

### 1.3 与现有工具的差异化

| 工具 | 定位 | APIWatch 的不同 |
|---|---|---|
| **OpenTelemetry + Jaeger/Zipkin** | 功能强大的分布式 tracing | 接入重、需跑后端；APIWatch 本地零后端、一行接入、开箱即用 |
| **Datadog / New Relic / Sentry** | 云端、面向生产环境、收费 | APIWatch 本地、免费、面向**开发态** |
| **django-debug-toolbar / flask-debugtoolbar** | 单框架、单进程内嵌 | APIWatch **跨框架统一**、独立 collector、可聚合看板、未来跨语言 |

**空白点**：「本地运行 + 零后端 + 多框架统一 + 一行接入 + 开发态聚焦」这个组合目前没有主流工具正面覆盖。

### 1.4 架构总览

```text
业务应用(:8080) + ASGI探针  ──POST /events──▶  Collector(:8765)  ──▶  Dashboard(:8765/dashboard)
        │                    统一JSON契约         SQLite + 聚合            原生HTML/JS
        └─(可选B) mount_dashboard="/APIWatch" 把看板挂到 :8080/APIWatch
```

**唯一扩展契约**：探针与 collector 之间只传符合 `spec/event.schema.json` 的 JSON，`POST /events`。任何语言只要产出该 JSON 即可接入——collector 与 dashboard 与语言无关。

### 1.5 看板两种部署模式

| 模式 | 看板地址 | 说明 |
|---|---|---|
| **A（默认）** | `127.0.0.1:8765/dashboard` | 独立 collector 服务，与业务解耦，可同时观测多个项目；collector 崩了不影响业务 |
| **B（可选开关）** | `业务应用端口/APIWatch` | 一个端口全搞定，符合「在自己应用里看数据」的直觉，`mount_dashboard="/APIWatch"` 启用 |

### 1.6 技术选型

- **探针（Python）**：运行时零强依赖，上报走标准库 `urllib` + 后台线程队列。
- **Collector**：FastAPI + 标准库 `sqlite3`（依赖 fastapi / uvicorn / pydantic）。
- **Dashboard**：原生 HTML/JS 单文件，零构建、零 Node 依赖。
- **trace 标识**：W3C Trace Context 标准 `traceparent`。

## 二、已完成的内容（MVP1，已通过验证）

MVP1 目标：不做 VSCode 插件，只做「命令行 + 本地 dashboard」的最小可运行闭环——**业务应用（接探针）→ collector（存储+聚合）→ dashboard（可视化）**。

### 2.1 交付清单

```text
APIWatch/
├─ spec/                          # ✅ 语言中立的事件契约（一等公民）
│   ├─ SPEC.md                    #    契约规范：字段、/events 接口、trace 规则、版本演进
│   └─ event.schema.json          #    版本化 JSON Schema（当前 1.0）
├─ probes/python/apiwatch/        # ✅ Python ASGI 探针
│   ├─ core/
│   │   ├─ event.py               #    统一事件模型 ApiEvent
│   │   ├─ trace.py               #    trace_id/span_id/traceparent 生成与解析
│   │   ├─ client.py              #    fire-and-forget 异步上报客户端
│   │   └─ config.py              #    配置（collector_url/project/enabled）
│   ├─ integrations/asgi.py       #    ApiWatchASGIMiddleware（FastAPI/Litestar 通用）
│   └─ pyproject.toml
├─ collector/apiwatch_collector/  # ✅ 语言无关：FastAPI + SQLite + 聚合
│   ├─ app.py                     #    FastAPI 路由 + CORS
│   ├─ storage.py                 #    SQLite 存储层（线程安全）
│   ├─ aggregate.py               #    summary/apis 聚合 + P95
│   ├─ models.py                  #    pydantic 契约模型
│   ├─ cli.py                     #    apiwatch start 命令入口
│   └─ dashboard/index.html       #    原生 HTML/JS 看板
├─ examples/fastapi_demo.py       # ✅ 端到端示例（正常/慢/报错三类接口）
├─ README.md                      # ✅ 定位/接入/差异化/路线
└─ 项目笔记.md                     #    本文件
```

### 2.2 已实现的能力

- **事件契约（spec）**：语言中立、版本化的 JSON Schema，冻结全部请求级字段，声明「未来 collector 换语言重写也须兼容此接口体」。
- **探针（ASGI 中间件）**：
  - 纯 ASGI 实现（不 import fastapi/starlette 具体类），FastAPI 与 Litestar 通用；
  - 采集 method / path / status / 耗时；route **尽力采、采不到用 path 兜底**；
  - 捕获未捕获异常 → 记录 `error_type/error_message` + `status=500` → 上报后**原样抛出，不吞异常**；
  - **fire-and-forget** 上报：后台线程 + 队列，队列满丢最旧，collector 不可用静默丢弃，**绝不影响业务请求**；
  - 入站 `traceparent` 合法则复用其 trace-id，为未来分布式串联铺路；
  - **B 模式开关** `mount_dashboard`：把看板挂到业务应用端口下。
- **Collector**：接收单条/批量事件写入 SQLite；提供 `/summary`、`/apis`、`/requests`、`/requests/{trace_id}`、`/dashboard`；启用 CORS；`apiwatch start` CLI 启动。
- **Dashboard**：指标卡（总请求/平均耗时/P95/错误率/错误数）、接口聚合表、慢接口 Top 5、错误接口排行、可展开 trace 详情的最近请求列表，每 3 秒轮询刷新。

### 2.3 验证结果

- **单元测试 45/45 全部通过**（`45 passed in 0.46s`），仅 1 个无害的 starlette 弃用警告。
- 覆盖：探针 trace 生成解析、ASGI 中间件事件组装/异常路径/route 兜底/traceparent 复用/非 http 透传；collector 的 SQLite 存储、聚合纯函数、以及经 `TestClient` **走真实 HTTP** 的全部 API 端点（单条/批量上报、summary/apis/requests 结构、trace 详情、错误计数）。
- 已完成两个包的可编辑安装（`pip install -e`），依赖 fastapi 0.139 / uvicorn 0.51 / pydantic 2.13 就位。

### 2.4 本地运行方式

```bash
# 1. 安装（探针 + collector 两个包）
pip install -e probes/python
pip install -e collector

# 2. 启动本地 collector（看板 http://127.0.0.1:8765/dashboard）
apiwatch start

# 3. 另开终端跑示例应用（:8080），再打几个请求
python examples/fastapi_demo.py
curl http://127.0.0.1:8080/api/users/1    # 正常
curl http://127.0.0.1:8080/api/slow       # 慢接口
curl http://127.0.0.1:8080/api/boom       # 故意报错
```

## 三、MVP2 已完成的内容

MVP2 目标：在 MVP1 闭环基础上补齐 Python 主流框架接入，并把 dashboard 从“最近请求展开”增强为“单请求详情”。

### 3.1 交付清单

```text
probes/python/apiwatch/core/capture.py          # ✅ 共享请求采集上下文与事件构造
probes/python/apiwatch/integrations/flask.py    # ✅ Flask 集成 ApiWatchFlask
probes/python/apiwatch/integrations/django.py   # ✅ Django 标准 middleware
examples/flask_demo.py                          # ✅ Flask 端到端示例
collector/apiwatch_collector/dashboard/index.html # ✅ 单请求详情视图
probes/python/tests/test_flask.py               # ✅ Flask 集成测试
probes/python/tests/test_django.py              # ✅ Django 集成测试
```

### 3.2 已实现的能力

- **Flask 集成**：`ApiWatchFlask(app, project="demo")` 或 `ApiWatchFlask().init_app(app)`，通过 `before_request` / `after_request` / `teardown_request` 采集请求、状态码、异常和 route pattern。
- **Django 集成**：标准 middleware `apiwatch.integrations.django.ApiWatchDjangoMiddleware`，从 Django settings 读取 `APIWATCH_PROJECT` / `APIWATCH_COLLECTOR_URL` / `APIWATCH_ENABLED`。
- **框架字段完善**：ASGI 示例显式上报 `fastapi`，Flask 上报 `flask`，Django 上报 `django`。
- **route pattern 增强**：Flask 使用 `request.url_rule.rule`；Django 使用 `request.resolver_match.route`，失败时仍由 collector 用 path 兜底聚合。
- **请求详情页**：dashboard 点击最近请求后调用 `/requests/{trace_id}`，展示 project、framework、method、path、route、status、duration、timestamp、trace_id、span_id、traceparent、错误信息。
- **探针依赖策略**：核心包仍保持运行时零强依赖；Flask/Django 通过 optional extras 安装：`probes/python[flask]`、`probes/python[django]`、`probes/python[all]`。

### 3.3 验证结果

- **自动化测试 63/63 全部通过**（`63 passed in 0.73s`），仅 1 个 starlette/httpx 相关弃用警告。
- 覆盖：MVP1 原有 ASGI/collector/storage/aggregate 测试，新增 Flask 正常/404/异常传播/500 响应/traceparent 复用，新增 Django 正常/异常/traceparent 复用/禁用采集。

### 3.4 MVP2 稳定化增强

- **CLI 维护命令**：新增 `apiwatch doctor` 检查 collector 可用性；新增 `apiwatch clear` 清空事件数据，支持 `--project`。
- **Collector 筛选接口**：`/summary`、`/apis`、`/requests` 支持 `project` / `framework` 查询参数；新增 `/filters` 返回 project/framework 选项；新增 `DELETE /events` 清空数据。
- **Dashboard 演示体验**：新增 project/framework 筛选、手动刷新、清空数据按钮；保留单请求详情。
- **三框架 demo**：新增 `examples/django_demo.py`，与 FastAPI/Flask 示例保持正常、慢、报错三类接口。
- **发布记录**：新增 `CHANGELOG.md`，记录 MVP1、MVP2 与稳定化增量。

## 四、MVP3 已完成的内容

MVP3 目标：把 APIWatch 的本地 collector 与接入指引纳入 VSCode 工作流，让用户不用离开编辑器即可启动采集、打开看板和查看框架接入方式。

### 4.1 交付清单

```text
vscode-extension/package.json             # ✅ VSCode command/config contribution
vscode-extension/src/extension.ts         # ✅ 扩展入口与命令注册
vscode-extension/src/collector.ts         # ✅ collector 启停与 doctor
vscode-extension/src/frameworkDetector.ts # ✅ Python Web 框架识别
vscode-extension/src/integrationGuide.ts  # ✅ 接入指引生成
vscode-extension/src/test/*.test.ts       # ✅ 插件纯逻辑测试
```

### 4.2 已实现的能力

- **命令面板入口**：`APIWatch: Start Collector`、`Stop Collector`、`Open Dashboard`、`Show Integration Guide`、`Doctor`。
- **Collector 进程管理**：插件通过配置的 Python 可执行文件运行 `python -m apiwatch_collector.cli start`；如果 collector 已经可访问则复用，不重复启动；停止命令只停止插件启动的进程。
- **Dashboard 打开**：默认外部浏览器打开 `http://127.0.0.1:8765/dashboard`；配置 `apiwatch.openDashboardInWebview` 可改为 Webview。
- **框架识别**：扫描 `pyproject.toml`、requirements、`manage.py` 和 Python 源码，识别 FastAPI / Litestar / Flask / Django。
- **接入指引**：根据识别出的框架生成 Markdown 代码片段，不自动修改用户业务代码。
- **插件配置**：`apiwatch.collectorHost`、`apiwatch.collectorPort`、`apiwatch.pythonExecutable`、`apiwatch.openDashboardInWebview`。

### 4.3 验证结果

- VSCode 插件 `npm run compile` 通过。
- VSCode 插件 `npm test` 通过，覆盖框架识别和接入指引生成。

## 五、未来准备做的内容

### 5.1 MVP4 —— 增强诊断能力

- **请求内多 span 瀑布图（粒度 2）**：埋点 DB 驱动、HTTP 客户端等，展示一次请求内部各段耗时。
- 慢接口检测、错误接口排行强化。
- **P99** 指标、`traceparent` 响应头。
- 简单诊断建议。

### 5.2 多语言探针扩展（长期方向）

- 按语言分层新增 `probes/go/`、`probes/node/` 等，各自实现一套探针，**遵守同一份 `spec/` 契约**。
- 验证「**collector 与看板零改动**」的可扩展承诺——扩展一门新语言 = 只需新写一个探针产出符合契约的 JSON。

### 5.3 更远期的探索

- **分布式 tracing 跨服务串联**：基于已生成/复用的 W3C `traceparent`，把多服务的调用链关联起来。
- **前后端操作关联**：前端注入 traceparent + 后端透传，把「前端一次操作」关联到「后端 API 请求」——最有差异化价值的亮点，单独立项。

## 六、软著方向（备选）

该项目形态独立可运行、技术创新点清晰（多框架统一探针 + 本地遥测采集 + 可视化诊断），适合作为软著主题。可选名称：

- 开发态接口链路追踪与性能分析软件
- 多框架 API 调用遥测采集与可视化系统

**功能描述**：本软件面向 Python Web 开发环境，支持 Django、Flask、FastAPI、Litestar 等框架，通过统一请求探针采集 API 调用数据，生成链路标识，记录状态码、耗时、错误、调用频次等指标，并通过本地采集服务和可视化看板提供接口性能分析、错误定位和开发态诊断能力。
